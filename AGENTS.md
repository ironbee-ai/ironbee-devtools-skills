# Agent Guidelines for ironbee-devtools-skills

> **Keep this file up to date.** When you change conventions, structure, or the relationship to the upstream `ironbee-devtools` project, update the relevant sections here as part of the same change.
> This file is the source of truth for AI coding agents working on this skills repository.

## What this project is

`ironbee-devtools-skills` ships skills.sh-installable skills for [IronBee DevTools](https://github.com/ironbee-ai/ironbee-devtools). Each skill is a Markdown file (plus a `references/` folder for CLI skills) that AI coding agents — Claude Code, Cursor, Windsurf, etc. — load to drive the IronBee DevTools CLIs.

This repo is **purely documentation / agent prompts**. There is no source code, no build step, and no runtime. Every change here is content edits to Markdown files, and correctness is judged by:
1. Whether the CLI examples actually run against the IronBee DevTools binaries.
2. Whether the documented flag names and input shapes match the tool schemas in the upstream `ironbee-devtools` project.
3. Whether the cross-references between SKILL.md files and `references/*.md` files resolve.

## Upstream source of truth: `ironbee-devtools`

**This repo is downstream of `ironbee-devtools`.** Every CLI subcommand, every flag, every default value, every domain folder, every MCP server-instructions paragraph — all of that is generated by code in the upstream repo. If you change anything here, verify it against the upstream tool definitions, never the other way around.

Specifically:
- A tool's CLI subcommand name comes from its `name()` method: `domain_command-name` → CLI `domain command-name`. Tools whose name has no underscore (e.g. `scenario-add`, `execute`) fall under the auto-generated `default` domain — **with two exceptions**: `execute` is special-cased to `run execute` in the CLI runner, and `scenario-run` is registered only in the daemon's `ToolRegistry` (not in `cliProvider.tools`), so it has **no CLI subcommand at all** — invoke it via `cli run execute --code "await callTool('scenario-run', { name: '...' })"` or from any MCP client.
- A tool's allowed CLI flags come from its Zod `inputSchema()`: every camelCase field becomes a kebab-case flag. Object and array fields stay as **single JSON-valued flags** (`--params '[...]'`), never split into per-key flags. Boolean fields typed `z.boolean().optional()` (without `.default(true)`) become a single switch — passing it sends `true`, omitting it sends `undefined` and the server applies its default. **`--no-<flag>` is NOT auto-registered**; only `z.boolean().default(true)` wraps generate both `--flag` and `--no-flag`. To force `false` from CLI on these fields, call via `run execute` / MCP.
- A tool's description in the skill prose should match the substance of `description()` in code, not invent new behavior.
- The CLI binary names live in upstream `package.json`'s `bin` block: `ironbee-devtools-cli` (browser default, alias `ironbee-browser-devtools-cli`), `ironbee-node-devtools-cli`, `ironbee-backend-devtools-cli`. Never use the legacy `browser-devtools-cli` / `node-devtools-cli` / `browser-devtools-mcp` names.
- The default daemon port is `2020` for all three CLIs. When multiple daemons run simultaneously the examples should pick different `--port` values (browser 2020, node 2021, backend 2022 by convention).

When in doubt, read the actual `name()` / `description()` / `inputSchema()` of the upstream tool — that is the contract.

### CLI skill vs task-specific skill — when to add which

**CLI skill** (`ironbee-*-devtools-cli/`): one per platform binary. Mirrors the upstream tool surface 1:1. Has a `SKILL.md` with install / global options / domain table / examples, plus a `references/` folder where each file documents one domain.

**Task-specific skill** (e.g. `browser-testing/`, `backend-testing/`): one per recurring workflow. Composes tools from one or more CLI skills into a recipe (e.g. "log-correlated request → assertion"). No `references/` folder. Lives at `skills/<name>/SKILL.md`.

When adding a new domain to upstream, the change goes into the **CLI skill** (new ref file + table row in SKILL.md). Add a new task skill only when a complete user workflow doesn't fit any existing one.

## Skill file conventions

### Frontmatter
Every `SKILL.md` starts with YAML frontmatter:

```markdown
---
name: <kebab-case-folder-name>
description: <one-line agent-facing description; this is what the model sees when picking a skill>
allowed-tools: Bash(ironbee-<binary>:*)[, Bash(ironbee-<other-binary>:*)]
---
```

- `name` MUST exactly match the folder name.
- `description` is the agent-facing trigger text. Be specific about which tools / scenarios this skill covers and (when relevant) which it does NOT cover — agents pick a skill by reading this line.
- `allowed-tools` lists only the binaries this skill needs. The browser CLI alias `ironbee-devtools-cli` is implied by `ironbee-browser-devtools-cli` — add it explicitly only if your examples use the short form.

### Cross-references between SKILL.md files
Use relative paths and the folder names exactly: `[../ironbee-backend-devtools-cli/SKILL.md]`. The skill checker validates these links exist.

### CLI examples
Every CLI example in a `SKILL.md` or a `references/*.md` file must use:
- the correct binary name (no legacy `browser-devtools-cli` / `node-devtools-cli` / `browser-devtools-mcp`)
- only flags that exist on the tool — verified against `skills/CHECK_PLAN.md` §1–§3
- single JSON-valued flags for object/array params — `--limit '{"count":10}'`, NOT `--limit-count 10`
- boolean flags as the flag alone (`--full-page`, not `--full-page true`)

The forbidden patterns list is canonical: `skills/CHECK_PLAN.md` §4.

### Reference (`references/<domain>.md`) tables
Each tool gets a flag table in this exact shape:

```markdown
| Argument | Type | Required | Default | Description |
|----------|------|----------|---------|-------------|
| `--<flag>` | <type> | Yes/No/Conditional | <default> | <description> |
```

- Types: `string`, `number`, `boolean`, `enum`, `object`, `array`.
- For object/array params, the `Type` column says `object` or `array` and the description should hint at the JSON shape (e.g. `JSON: {kind:'json', value:<any>}`).
- For mutually-exclusive params (e.g. `--code` vs `--file`), mark each `Conditional` and add a footnote.

### Naming consistency
Always say "ironbee-devtools" / "IronBee DevTools" — never "browser-devtools-mcp", "browser-devtools-cli", or "node-devtools-cli" (these are the pre-rebrand names and are listed as forbidden in `CHECK_PLAN.md` §4).

## The CHECK_PLAN.md contract

`skills/CHECK_PLAN.md` is the **only** place where the full mapping from tool name → allowed CLI flags lives in this repo. It is the agent-runnable contract for the skill checker:

- §1 lists browser tools and their flags.
- §2 lists node tools and their flags.
- §3 lists backend tools and their flags.
- §4 lists forbidden patterns (legacy names, split-up object flags, scroll's `--direction`, etc.).
- §5 maps `references/<domain>.md` files to the tools they document.
- §6 is the check procedure (run the forbidden grep, validate refs against §5, validate flag tables against §1–§3, run 3 passes).

**When upstream `ironbee-devtools` adds, renames, or removes a tool or flag, update CHECK_PLAN.md FIRST, then the corresponding `references/*.md` and the SKILL.md examples.** That ordering keeps CHECK_PLAN.md authoritative.

## Workflows

### Adding a new tool / domain that exists upstream
1. Read the upstream tool's `name()`, `description()`, `inputSchema()`.
2. Decide which CLI skill it belongs to (browser / node / backend) based on which platform's `tools/` folder it lives in upstream.
3. Update `skills/CHECK_PLAN.md` — add a row in §1/§2/§3, and (if it's a new ref file) add it to §5.
4. Update or create the matching `references/<domain>.md`. Use the table format above. Translate every camelCase input field to a `--kebab-case` flag.
5. Update the parent `SKILL.md` — add to the domain table, add an example if the tool is high-traffic.
6. Run the CHECK_PLAN procedure end-to-end (3 passes) before considering the change done.

### Renaming or removing an upstream tool/flag
1. Update `CHECK_PLAN.md` first (remove the row, or rename it).
2. Search the repo (`grep -r '<old-name>' skills/`) for every mention and fix them.
3. Run the forbidden-grep step to catch leftovers.

### Adding a new task-specific skill
1. Pick a folder name that describes the workflow, not the tools (e.g. `backend-testing`, not `http-grpc-skill`).
2. Frontmatter: write the agent-facing `description` so the model knows when to trigger this skill vs the closest neighbor (call out the boundary explicitly — see `api-testing` vs `backend-testing` for the pattern).
3. Body: open with "When to Use" bullets, then a flat list of capabilities (each a small CLI snippet), then 1–3 canonical end-to-end flows, then a "Best Practices" list.
4. Cross-link to the CLI skill that hosts the underlying tools.

### Updating from upstream changes (the common case)
The user's typical message is "ironbee-devtools'ta X değişti, skills'i de güncelle". For that:
1. Confirm with the user whether the change is purely internal (no skill update needed — refactor, private helpers, underscore-prefixed internal fields) or surface-affecting.
2. If surface-affecting, read the upstream diff and locate the affected tool/domain.
3. Update CHECK_PLAN.md → `references/*.md` → `SKILL.md` (in that order).
4. Re-run the CHECK_PLAN procedure.

## Conventions

### Markdown style
- Headings: ATX-style (`#`, `##`, `###`). One `#` per file.
- Code fences: triple backticks with a language hint (`bash`, `json`, `markdown`).
- Tables: pipe-and-dash; align column widths visually if it improves readability.
- Bold for **defined terms** and **emphatic instructions**, italics sparingly.
- Never use emojis in skill content unless the user explicitly asks.

### Language
- Skills are agent-facing first, human-readable second. Write declaratively, in the imperative: "Pin a trace id" not "You can pin a trace id".
- Avoid filler / hedging ("typically", "usually", "in most cases") unless it really is a tendency, not a rule.
- When describing behavior the agent has to mirror (defaults, precedence, redaction lists), be explicit: name the env var, list the keys.

### File layout
- One blank line between top-level sections, no trailing whitespace.
- Long flag tables: keep the column order `Argument | Type | Required | Default | Description`. Do not reorder.

### What NOT to do
- Do not add CLI flags that don't exist on the upstream tool. Verify in `inputSchema()` first.
- Do not split object/array params into multiple flags ("`--limit-count`" / "`--header-x`") — those are forbidden patterns and the upstream CLI rejects them.
- Do not introduce a new SKILL.md without giving it a meaningful `description` field; it will be invisible to the model otherwise.
- Do not write tutorials. Skills are reference + recipes, not blog posts.
- Do not commit local paths, internal usernames, or any environment-specific paths from `WORKING_DIR` etc.

## Installation context (for the agent's own awareness)

End users install this skill bundle via:

```bash
npx skills add ironbee-ai/ironbee-devtools-skills
```

The CLI binaries themselves come from the npm package:

```bash
npm install -g @ironbee-ai/devtools
```

These two commands appear in every CLI skill's "Installation" section and the top-level `README.md`. Keep them consistent across files.
